<summary>🧭 How to read this report (framing & method)</summary>

**What this is ultimately measuring.** When you hand work to an AI here, does it behave like a brand-new hire or like an engineer who has been in the org eighteen months? The difference isn't capability - it's *externalized context*: knowing where things live, which contracts are load-bearing, where the minefields are. An AI contributor is structurally always the new hire - it starts every session with an empty head, seeing one narrow slice through its context window - so this report scores how much of the tenured engineer's implicit map the codebase has made explicit and navigable. The aim is not an AI that comprehends what humans no longer can and is trusted blindly (that is the dangerous case); it is a codebase legible enough that the relevant slice fits one context window and the agent's answers stay anchored to code you can still verify. **Legibility you can trust, not omniscience you can't.** The write-side layers (linters, architecture tests, CI gates, coverage, review automation) read not as a leash on the agent but as the guardrails that protect any contributor, human or AI, from costly mistakes by design. A **Missing** there means the codebase asks the contributor to be careful where it could instead make the wrong move hard to make.

This is an improvement roadmap, not a verdict. It pairs three views:

- **Where the codebase is today** - the complexity heatmap. Vivid red = complex AND actively changing = the files most likely to bite next week.
- **Whether an agent can navigate it** - the doc graph: how much is reachable from the entry point, and which docs are stale maps of churning code.
- **What keeps it from getting worse** - the AI Readiness score (0-8) across read-side foundation, write-side enforcement, and meta.

A codebase can be 8/8 and still on fire, or 2/8 with a calm treemap. The views matter together.

**The legacy-transition lens - this is not a new problem.** The hard part of AI-assisted coding is the same one the industry has faced for 25 years: *what happens when the people who understood the code are gone, or the code outruns anyone's ability to understand it?* AI only changes the **velocity**. Read this report through the accumulated discipline of legacy-code engineering:

- **"Legacy code is code without tests."** (Feathers, 2004) A complex, churning, untested file is already legacy. Write **characterization tests** to pin actual current behaviour, find a **seam**, *then* refactor.
- **Don't rewrite - strangle.** (Spolsky, 2000; Fowler's Strangler Fig) An agent's default ("regenerate from intent") discards undocumented accreted behaviour. Replace incrementally behind tests.
- **Hotspots and change-coupling are the map.** (Tornhill / CodeScene) The complexity-×-churn heatmap and `hidden_coupling` seams are the prioritized worklist.
- **Decide per component (the 7 Rs).** Dead weight → retire; complex-and-live → characterize then refactor; `refactor_boundary` zones → hand to an agent in isolation.

**How it's measured.** An AI-readiness review run almost entirely on *traditional* tooling - static analysis, git history, and graph metrics. The model only writes the prose around those numbers; it does no scanning itself. That keeps a run fast, near-zero in model tokens, and reproducible run-to-run.
