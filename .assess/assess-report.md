<summary>🧭 How to read this report (framing & method)</summary>

**What this is ultimately measuring.** When you hand work to an AI here, does it behave like a brand-new hire or like an engineer who has been in the org eighteen months? The difference is *externalized context*: knowing where things live, which contracts are load-bearing, where the minefields are. An AI contributor is structurally always the new hire - it starts every session with an empty head, seeing one narrow slice through its context window - so this report scores how much of the tenured engineer's implicit map the codebase has made explicit and navigable. The aim is legibility you can trust, not omniscience you can't. The write-side layers (linters, architecture tests, CI gates, coverage, review automation) are not a leash on the agent - they are the guardrails that make a costly mistake hard to make for any contributor, human or AI.

This is an improvement roadmap, not a verdict. It pairs three views: **where the codebase is today** (the complexity heatmap - vivid red = complex AND churning), **whether an agent can navigate it** (the doc graph - reachability and lying maps), and **what keeps it from getting worse** (the 0-8 score across read-side, write-side, and meta bands). A codebase can be 8/8 and still on fire, or 2/8 and calm - the views matter together. For Meridian the unusual reading is 8/8 *with* a live keyhole list: the scaffolding is complete and the residual risk is purely documentation truth.

**The legacy-transition lens.** AI only changes the velocity of an old problem - a codebase can slide from modern to legacy in months because agents generate change faster than a team keeps comprehending it. Read the findings through accumulated legacy-code discipline: a complex, churning, untested file is already legacy (Feathers - write **characterization tests** to pin current behaviour, find a **seam**, *then* refactor); don't rewrite, **strangle** (Fowler's Strangler Fig over Spolsky's never-rewrite); the complexity-×-churn heatmap and coupling findings *are* Tornhill's (CodeScene) hotspot analysis; decide per component with the **7 Rs**. None of Meridian's findings is a refactor task - they are doc-truth and test-independence actions - so the precedent that applies here is "a wrong map is worse than none", not "rewrite the hotspot".

**How it's measured.** Almost entirely traditional tooling - static analysis, git history, graph metrics over docs and code. The model only writes the prose around the numbers; it does no scanning itself. That keeps a full run fast, near-zero in model tokens, and reproducible run-to-run.
