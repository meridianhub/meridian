# Service README Template

**Status:** Active
**Audience:** Engineers, AI contributors, integrators
**Scope:** Per-service `README.md` under `services/<service>/`

Every Meridian service must publish a `README.md` that follows the structure below.
The template exists so AI contributors and new engineers can locate the same
information at the same offset in every service. Deviations are allowed only when a
section is genuinely not applicable, in which case the section heading remains and
its body explains why.

## Required Sections

A compliant service README has the following sections in this order:

1. Frontmatter
2. Overview (table)
3. API Surface
4. Domain Model
5. Dependencies
6. Dependents
7. Load-Bearing Files
8. Configuration

Additional sections (Architecture diagrams, Health Endpoints, Local Development,
Security Considerations, References) are encouraged but optional. They follow the
required sections.

---

## 1. Frontmatter

YAML frontmatter at the top of the file, between `---` fences. Used by the skills
system to surface the service when a relevant trigger appears in conversation.

```yaml
---
name: <service-name>
description: <one-sentence description of what the service does>
triggers:
  - <situation in which this service is the right answer>
  - <another situation>
instructions: |
  <multi-line guidance for AI contributors working on this service. Cover
  protocols, ports, conventions, and any non-obvious rules. Keep it short -
  the README body is the long form.>
---
```

The `name`, `description`, `triggers`, and `instructions` fields are mandatory.
Use lowercase kebab-case for `name`. Triggers should describe situations, not
keywords.

## 2. Overview

A single markdown table summarising the service. The exact rows are:

```markdown
## Overview

| Attribute | Value |
|-----------|-------|
| **BIAN Domain** | <BIAN service domain or "Infrastructure (non-BIAN)"> |
| **Layer** | <layer name from docs/architecture-layers.md> |
| **Port** | <gRPC port> (gRPC), <HTTP port> (HTTP) |
| **Database** | <CockroachDB schema name> or N/A |
| **Standalone** | Yes / No (list service dependencies) |
```

Layer must match exactly one of the layer names defined in
[`docs/architecture-layers.md`](architecture-layers.md). If the service spans no
BIAN domain (gateways, infrastructure), set `BIAN Domain` to
`Infrastructure (non-BIAN)`. Standalone is `No` for any service that requires a
peer Meridian service at runtime.

## 3. API Surface

The externally observable contract for the service. Use one or both of the
following formats depending on the protocol:

### gRPC services

```markdown
| Service | RPC | Purpose |
|---------|-----|---------|
| `MyService` | `DoThing` | One-line description |
```

### HTTP endpoints

```markdown
| Method | Path | Purpose |
|--------|------|---------|
| `POST` | `/v1/things` | One-line description |
```

For Edge services that proxy other services, list the protocols supported at
the perimeter and link to the per-service tables of the downstream targets.
Do not duplicate downstream RPC lists in the Edge README.

## 4. Domain Model

A Mermaid `classDiagram` showing the principal entities owned by the service
and their relationships. Use Mermaid over ASCII art. If the service is a
stateless adapter (gateway, transcoder, MCP server), this section may carry a
short note explaining there is no domain model and a link to the upstream
service whose entities it surfaces.

```markdown
## Domain Model

\`\`\`mermaid
classDiagram
    class Instruction {
        +UUID id
        +string type
        +InstructionStatus status
        +time createdAt
    }
    class ProviderConnection {
        +UUID id
        +string provider
        +ConnectionStatus status
    }
    Instruction --> ProviderConnection : routed via
\`\`\`
```

## 5. Dependencies

Services this one calls. Direction is outbound. List each dependency with the
protocol used and a one-line rationale.

```markdown
| Service | Protocol | Purpose |
|---------|----------|---------|
| `position-keeping` | gRPC | Balance reads per ADR-0023 |
| `reference-data` | gRPC | Instrument validation |
```

Include infrastructure dependencies (CockroachDB, Kafka, Redis) only when they
are not present in every Meridian service. Universal infrastructure goes in
Configuration.

## 6. Dependents

Services that call this one. Direction is inbound. Generated by grepping for
proto imports and gRPC client constructors across the codebase. List as a table
with the entry point on the caller side so reviewers can audit the link.

```markdown
| Service | Entry Point | Purpose |
|---------|-------------|---------|
| `payment-order` | `services/payment-order/adapters/gateway/fg_client.go` | Dispatches payments |
```

Keep this list current. A stale Dependents list defeats the navigability goal
of the template.

## 7. Load-Bearing Files

Files that a contributor must read before changing the service. These are not
necessarily the largest files - they are the ones whose contracts other code
depends on, or whose invariants are subtle. Typical candidates:

- `cmd/main.go` (process wiring)
- `service/server.go` (gRPC server implementation)
- gRPC handler files
- Auth middleware
- Routing configuration
- Saga state machines
- Domain entities with lifecycle invariants

Format:

```markdown
| File | Why It Matters |
|------|----------------|
| `cmd/main.go` | Wires every component; changes affect startup order |
| `service/server.go` | Implements the gRPC contract; signature changes break callers |
```

Paths are relative to the service directory.

## 8. Configuration

Environment variables consumed by the service. Group by concern (core,
authentication, observability) when the list is long.

```markdown
| Variable | Required | Default | Purpose |
|----------|----------|---------|---------|
| `DATABASE_URL` | Yes | - | CockroachDB connection string |
| `GRPC_PORT` | No | `50063` | gRPC listen port |
```

Document only variables the service actually reads. A variable referenced by
Kubernetes manifests but ignored by the binary belongs in the deployment docs,
not here.

## Optional Sections

The following sections may appear after Configuration when they add value:

- **Architecture** - Mermaid flowchart of internal components if the service is
  non-trivial in shape (e.g., an Edge transcoder with multiple middleware
  layers).
- **Health Endpoints** - When the service exposes non-standard health probes.
- **Local Development** - Tilt instructions, standalone build instructions,
  test invocations.
- **Security Considerations** - When the service handles auth, secrets, or
  signed payloads.
- **References** - Links to ADRs, related services, and external specs.

## Authoring Notes

- No emojis in any service documentation.
- Use Mermaid for diagrams; never ASCII art for boxes and arrows.
- Use markdown tables (`| col | col |`) and never ASCII tables.
- Cite the canonical Layer name from
  [`docs/architecture-layers.md`](architecture-layers.md) verbatim.
- When in doubt about which side of a dependency a service sits on, run
  `rg "<service>:50051" services/` and `rg "<service>v1" services/` to find the
  callers, then list them in Dependents.
- All Markdown files must pass `markdownlint`. The pre-commit hook enforces this.
